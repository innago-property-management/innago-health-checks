# Delphi Review: Dynamic Health Checks

Three parallel panels reviewed the spec and design from distinct lenses. Findings are synthesized below with resolution guidance. Items marked **🚫 Blocker** must be resolved before coding begins. Items marked **⚠️ Pre-Ship** must be resolved before merging. Items marked **📝 Non-blocking** are improvements that can happen during or after implementation.

---

## Panel 1: Security & Credentials

### S1 — Vault-file fallthrough on parse failure contradicts the spec [🚫 Blocker]

**Severity: Critical**

`VaultSecretsReader.TryRead` catches JSON parse exceptions and returns `null`. Returning `null` causes `TryStrategy2VaultFileAsync` to fall through to Strategy 3 (IConfiguration). This directly contradicts SPEC.md §Error Handling: _"File read failures are treated as Unhealthy (do not fall through to IConfiguration strategy)."_

A corrupt vault file during a credential rotation would silently cause the health check to probe a completely different connection source, potentially returning `Healthy` while the service's real credentials have been revoked.

**Resolution:** Distinguish "file absent" from "file present but unreadable." `TryRead` should return a discriminated result (e.g., a `(Dictionary<string,string>? Secrets, bool FileWasPresent, Exception? Error)` tuple or a `VaultReadResult` record). The strategy method maps:
- File absent → fall through (return `null`)
- File present, parse error → return `Unhealthy(error.Message)` immediately, do not fall through
- File present, host key absent → fall through (this case is still acceptable; vault file contains secrets for a different service)

Update SPEC.md error table and DESIGN.md `VaultSecretsReader` contract to match.

---

### S2 — Vault file presence does not block IConfiguration fallback [🚫 Blocker]

**Severity: Critical**

Per the current design, if the vault file exists but the host key (`rabbit_host` / `db_host`) is absent, Strategy 3 fires. This means a service that is vault-managed (vault file exists) can silently fall back to IConfiguration credentials — connecting to a different host with potentially stale or default credentials and returning `Healthy`.

The presence of the vault file is a strong signal that this service is vault-managed. When the file exists but is incomplete, it indicates a partial write during rotation, not a deliberate "use IConfiguration instead" signal.

**Resolution:** Add a `bool _vaultFileWasPresent` gate: if the vault file exists at all, Strategy 3 must not be attempted. If vault file is present but yields no usable host, return `Unhealthy("Vault secrets file found but host key is absent.")`. Only when the vault file is genuinely absent does the fallback to Strategy 3 apply.

Update both SPEC.md and DESIGN.md to reflect this change.

---

### S3 — `guest/guest` default credentials silently authenticate [⚠️ Pre-Ship]

**Severity: Major**

`BuildFactory` defaults `UserName` and `Password` to `"guest"` when vault file keys are absent. If the vault file has `rabbit_host` but rotation dropped the credential keys (partial write), the health check will authenticate as `guest/guest`. Whether this succeeds depends on broker configuration, but the failure mode is invisible: the check may return `Healthy` with entirely wrong credentials.

**Resolution:** When resolving credentials from the vault file, if `rabbit_host` is present but credentials are absent, treat as `Unhealthy("Vault file present with host but missing credentials.")` — do not use defaults. Defaults are only appropriate for the IConfiguration fallback strategy where credentials are genuinely optional. Remove `?? "guest"` from the vault file code path.

---

### S4 — Raw exceptions attached to `HealthCheckResult` expose credentials [⚠️ Pre-Ship]

**Severity: Major**

Both `ProbeConnectionAsync` methods use `HealthCheckResult.Unhealthy(ex.Message, ex)`. The full exception object (including `InnerException`, `Data` dictionary, and stack trace) is attached. ASP.NET Core's development health response writer serializes the exception. Loki and OTel publishers may forward it. Npgsql and RabbitMQ.Client exceptions can include usernames, partial connection strings, or DSNs in their message chains.

**Resolution:**
```csharp
catch (Exception ex)
{
    _logger.ProbeConnectionFailed(ex);  // full exception to structured log only
    return HealthCheckResult.Unhealthy("Connection probe failed. See application logs for details.");
}
```
Log the full exception via `LogMessages` at `Warning` level. The health check result receives only the sanitized string — no exception reference. Add a README note that the attached exception is visible to `IHealthCheckPublisher` implementations.

---

### S5 — `NpgsqlConnectionStringBuilder.ConnectionString` includes plaintext password [⚠️ Pre-Ship]

**Severity: Major**

`NpgsqlConnectionStringBuilder` does not sanitize passwords when producing its `ConnectionString` property. If this string ever appears in a log message, exception message, or health check description (e.g., via S4's `ex.Message`), the password leaks into structured logs forwarded to Loki/OTel.

**Resolution:** Never log or embed the raw connection string in any user-visible string. When a connection string must appear in log context, build a sanitized version: clone the builder, set `Password = "[redacted]"`, use `.ConnectionString`. Add a `SanitizedConnectionString` internal helper used exclusively for diagnostic output.

---

### S6 — TOCTOU race on vault file read [📝 Non-blocking]

**Severity: Major**

`File.Exists` + `File.ReadAllText` is a TOCTOU pattern. Vault agent uses atomic file rename, but during the window between the two calls, the file may be briefly absent or empty. On network filesystems, atomicity is not guaranteed.

**Resolution:** Replace the two-step with a single `try { File.ReadAllText(...) } catch (FileNotFoundException) { return absent; }`. Optionally add a single retry with 50ms delay to tolerate atomic-rename windows. This is a well-understood pattern for vault-file consumers. Not blocking because the window is narrow, but should be done before shipping.

---

### S7 — `VaultSecretsPath` is unvalidated filesystem input [📝 Non-blocking]

**Severity: Minor**

Any absolute path can be supplied. Validate at registration time: must be absolute, no `..` segments, must end in `.json`. Throw `ArgumentException` at registration, not at probe time.

## Panel 2: Architecture & Testability

### A1 — `AddCheck` factory overload may not exist in 9.0.14 as written [🚫 Blocker]

**Severity: Major**

The PLAN Step 5e uses a factory lambda `sp => new RabbitMqHealthCheck(sp, ...)` with `builder.AddCheck(name, factory, tags)`. The standard `IHealthChecksBuilder.AddCheck` overloads do not include an `IServiceProvider`-aware factory. This pattern requires either `HealthCheckRegistration` constructor directly or a custom extension. The PLAN itself contains a self-correction note on this code, which is a signal the overload was not validated against the actual 9.0.14 API.

**Resolution:** At scaffolding time (Plan Step 1), validate the correct API path. The correct form is:
```csharp
builder.Services.Configure<HealthCheckServiceOptions>(opts =>
    opts.Registrations.Add(new HealthCheckRegistration(
        options.Name,
        sp => new RabbitMqHealthCheck(sp, sp.GetRequiredService<IConfiguration>(), options, sp.GetRequiredService<ILogger<RabbitMqHealthCheck>>()),
        failureStatus: null,
        tags: options.Tags)));
```
Update PLAN Step 5e and DESIGN.md DI Registration section before implementation begins.

---

### A2 — `sealed` + `internal virtual` = CS0549 compiler error under `TreatWarningsAsErrors` [🚫 Blocker]

**Severity: Major**

`virtual` on a method of a `sealed` class is CS0549, which becomes a compiler error under `TreatWarningsAsErrors=true` (active on Release builds). PLAN Step 5d declares both `internal sealed class` and `internal virtual async Task<HealthCheckResult> ProbeConnectionAsync`. The plan's own Step 7 proposes resolving this by removing `sealed` — but the resolution should be chosen before implementation, not discovered mid-build.

**Resolution (recommended):** Replace `internal virtual` + non-sealed with a constructor-injected `Func` delegate:
```csharp
internal sealed class RabbitMqHealthCheck : IHealthCheck
{
    private readonly Func<ConnectionFactory, CancellationToken, Task<HealthCheckResult>> _prober;

    internal RabbitMqHealthCheck(
        IServiceProvider serviceProvider,
        IConfiguration configuration,
        RabbitMqHealthCheckOptions options,
        ILogger<RabbitMqHealthCheck> logger,
        Func<ConnectionFactory, CancellationToken, Task<HealthCheckResult>>? prober = null)
    {
        _prober = prober ?? DefaultProbeAsync;
    }
}
```
The class stays `sealed`. Tests inject a lambda. No `InternalsVisibleTo` expansion needed for the override mechanism. No Moq subclassing required. Update DESIGN.md class diagram and PLAN Steps 5d, 7 before coding.

---

### A3 — `CreateChannelAsync` in strategies 2/3 is unnecessary and asymmetric [⚠️ Pre-Ship]

**Severity: Minor**

Strategy 1 checks `connection.IsOpen` only. Strategies 2 and 3 open a full connection plus a channel. This inconsistency means "healthy" is verified differently depending on which strategy fires. Opening a channel adds broker-side resource allocation on every probe interval and creates connection churn visible in broker metrics.

**Resolution:** Drop `CreateChannelAsync` from `ProbeConnectionAsync`. Check `connection.IsOpen` after `CreateConnectionAsync`. This aligns all three strategies' definition of "healthy" (broker is reachable and accepts credentials), and removes the channel overhead. Update SPEC.md strategy table footnote for strategies 2/3.

---

### A4 — `EnableConfigurationBindingGenerator=true` is cargo-culted and unused [📝 Non-blocking]

**Severity: Informational**

The new packages do not call `IConfiguration.Get<T>()` or `Bind()` on options types. The flag intercepts those calls to generate AOT-safe binding code, but with no such calls present, the generator emits nothing useful. It adds a build-time generator invocation and `obj/` noise from `EmitCompilerGeneratedFiles=true`.

**Resolution:** Remove `EnableConfigurationBindingGenerator=true` from both new package `.csproj` files. Add a comment near the STJ source-gen context: `// Vault JSON uses VaultSecretsJsonContext for AOT safety; no IConfiguration.Get<T>() calls in this assembly.`

---

### A5 — Plan schedules tests after production code — not TDD [📝 Non-blocking]

**Severity: Minor**

Steps 3-4 create empty test class files (scaffolding). Steps 5-6 implement all production code. Steps 7-8 implement test bodies. The CLAUDE.md global instruction is "ALWAYS favor TDD." The current plan ordering is code-first, tests-second.

**Resolution:** The executing coding agent should write each test method before the production method it covers. The plan step ordering can stay for scaffolding (steps 1-4), but steps 5-8 should interleave: for each class, write failing tests first, then implement until tests pass. Add a plan note making this explicit.

## Panel 3: Consumer & Migration

### C1 — `["ready"]` default tag unverified against live Kubernetes configs [🚫 Blocker]

**Severity: Critical**

The 18 services' Kubernetes probes filter `/health` by tag name. If any service uses a tag other than `"ready"`, and the consumer calls `.AddInnagoRabbitMq()` without specifying tags, the health check is invisible to Kubernetes. The pod will appear ready/live regardless of broker connectivity.

**Resolution:** Before implementation begins, audit the 18 unmerged feature branch PRs to extract the exact tag values used. If uniformly `"ready"`, the default is safe. If divergent, change the default to `null` / empty — forcing explicit tag specification — or add a startup log warning when `Tags` is at its default value: `"AddInnagoRabbitMq: using default tag 'ready' — verify this matches your Kubernetes probe configuration."` Consider overloads that make `tags` non-optional.

---

### C2 — Npgsql Strategy 1 key `"DefaultConnection"` likely wrong for Innago services [🚫 Blocker]

**Severity: Major**

Strategy 1 tries `GetConnectionString("DefaultConnection")` and `GetConnectionString("Database")`. Vault-managed Innago services almost certainly do not store connection strings under these scaffolded-template names. If the actual key name differs (e.g., `"Postgres"`, `"App"`, `"Primary"`), Strategy 1 silently returns `null` with no log output, falls through to vault/IConfiguration strategies, and the developer never knows Strategy 1 was skipped.

**Resolution:** Audit the 18 service repos for their actual Postgres connection string key names. Add a `ConnectionStringName` property to `NpgsqlHealthCheckOptions` (default: `null`). If set, use only that key. Log at `Debug` when Strategy 1 finds nothing: `"No connection string at 'DefaultConnection' or 'Database'; falling through to vault strategy."` Update the SPEC configuration keys table.

---

### C3 — Strategy 1 (DI IConnection) silently short-circuits on MassTransit services [⚠️ Pre-Ship]

**Severity: Major**

MassTransit >= 8 registers `IConnection` in the DI container. On MassTransit services, Strategy 1 will always fire, find MassTransit's connection, check `IsOpen`, and return `Healthy` — without the vault file or IConfiguration strategies ever running. A developer adding this package to a MassTransit service with broken vault config will see `Healthy` and not understand why.

**Resolution:** Add a `Debug`-level log in Strategy 1 naming the type of `IConnection` found: `"DI IConnection found (type: {TypeName}); using existing connection."` Add `IgnoreDiConnection` boolean to options (default `false`) to let vault-rotation services opt out of Strategy 1. Document the MassTransit IConnection behavior explicitly in the README.

---

### C4 — Duplicate health check name produces silent duplicate JSON keys [⚠️ Pre-Ship]

**Severity: Major**

ASP.NET Core's `AddCheck` does not throw on duplicate names — it produces two entries with the same key in the health report JSON. If a service migrating from the old copy-pasted code registers both (old code still in `Program.cs`, new package also registered), the health endpoint silently emits duplicate keys. JSON parser behavior for duplicate keys is undefined.

**Resolution:** In `AddInnagoRabbitMqCore`, inspect `builder.Services` for any existing `HealthCheckRegistration` with the same name and log a `Warning` if found: `"Health check named '{Name}' is already registered. Remove the manual registration to avoid duplicate entries."` Also document this in the migration guide.

---

### C5 — No migration transition guide [⚠️ Pre-Ship]

**Severity: Major**

The PLAN treats the 18-service migration as a clean atomic swap in a single PR. There is no guidance for the in-review state (both old and new code present), the two-PR strategy (ship new under a temporary name first, then remove old), or what happens if the PR reviewer merges before old code is removed.

**Resolution:** Add a `MIGRATION.md` to the spec directory (or package README section) covering: pre-migration audit steps, atomic-swap checklist, validation via `curl /health`, and the two-PR path. Include the explicit warning about duplicate names during the transition.

---

### C6 — MassTransit co-existence relies solely on README [📝 Non-blocking]

**Severity: Major**

SPEC US-6 defers `skipIfMassTransit` to future scope and relies on README documentation to prevent duplicate health check registration on MassTransit services.

**Resolution:** Add a startup-time warning log (not an error) when any service descriptor matching a MassTransit transport `IHealthCheck` is detected in `builder.Services`. This is a one-time log at registration, not a probe-time check. The README warning remains, but developers who miss it will see the warning in startup logs. The full `skipIfMassTransit` option stays deferred.

## Verdict

**HOLD on implementation.** The following must be resolved before coding begins:

1. **S1/S2** — Vault file present + parse error behavior (critical spec/implementation mismatch)
2. **A1** — Validate `AddCheck` factory overload against actual 9.0.14 API
3. **A2** — Resolve `sealed` + test seam design (choose Func delegate)
4. **C1** — Audit 18 service repos for actual tag names
5. **C2** — Audit 18 service repos for actual Postgres connection string key names

All five blockers are resolvable in a SPEC/DESIGN revision (round 2 of this Delphi, or direct updates). None require fundamental rethinking of the tri-strategy approach — the concept is sound, the implementation details need tightening.

The Security panel's S3, S4, S5 findings should be addressed during Step 5/6 implementation as concrete code decisions. The Migration panel's C4, C5 items should be completed as PLAN additions before implementation begins (they are doc work, not code work).
